# Messenger (PWA)

Свой мессенджер: 1-на-1 и групповые чаты (с добавлением/удалением участников и переименованием), поиск людей по юзернейму (@nick) и по ссылке-приглашению, обмен файлами и фото (с превью), статус "в сети"/"печатает", push-уведомления, устанавливается как приложение (PWA).

Также: загрузка своего аватара, редактирование и удаление сообщений, подгрузка истории по мере прокрутки (пагинация), счётчики непрочитанных сообщений, статусы "доставлено/прочитано" (галочки), архивирование чатов, отключение уведомлений (mute) и удаление чата из своего списка.

## Структура

- `backend/` — Node.js + Express + Socket.io + Prisma (PostgreSQL)
- `frontend/` — React + Vite + vite-plugin-pwa + Tailwind CSS

## Запуск локально

### 1. База данных

Создайте бесплатный PostgreSQL, например на [Neon](https://neon.tech) или [Railway](https://railway.app), и скопируйте `DATABASE_URL` (для прод-деплоя ниже используется именно Neon).

### 2. Backend

```
cd backend
npm install
```

Откройте `.env` и вставьте свой `DATABASE_URL` (сейчас там плейсхолдер). `JWT_SECRET` и `VAPID_*` ключи уже сгенерированы и готовы к использованию.

```
npx prisma migrate dev --name init
npm run dev
```

Если у вас уже есть база с более ранней версией этого проекта (в ней ещё нет миграций из папки `prisma/migrations`, либо она создавалась до добавления аватаров/статусов/архива), выполните ту же команду ещё раз с новым именем — Prisma сама досоздаст недостающие таблицы/колонки:

```
npx prisma migrate dev --name add_edit_read_archive
```

Backend поднимется на `http://localhost:4000`.

### 3. Frontend

```
cd frontend
npm install
npm run dev
```

Frontend поднимется на `http://localhost:5173` (запросы к `/auth`, `/conversations` и т.д. проксируются на backend).

## Проверено локально

- Регистрация (email + пароль + юзернейм) и вход
- Создание чата 1-на-1, обмен сообщениями в реальном времени через Socket.io между двумя аккаунтами
- REST-история сообщений при открытии чата, с пагинацией (подгрузка более старых сообщений при прокрутке наверх)
- Поиск людей строго по юзернейму (`@nick`): пустой ввод и поиск по имени/email ничего не возвращают, email нигде не отображается
- Ссылка-приглашение `/u/username` (включая редирект через `/login?next=...` для неавторизованных) открывает/создаёт чат с нужным человеком
- Групповые чаты: создание, переименование, добавление и удаление участников, выход из группы — с обновлением в реальном времени у остальных участников
- Отправка фото (инлайн-превью в чате) и обычных файлов (ссылка с оригинальным именем)
- Загрузка своего аватара из профиля — отображается везде (шапка, список чатов, сообщения) в реальном времени
- Редактирование и удаление собственных сообщений (удаление — мягкое, с плейсхолдером "Сообщение удалено" у всех участников)
- Счётчик непрочитанных сообщений в списке чатов, статусы "доставлено/прочитано" (галочки меняют цвет, когда собеседник открывает чат)
- Архивирование, отключение уведомлений (mute) и удаление чата из своего списка (у собеседника переписка остаётся; если он напишет снова, чат сам появится в списке)

Push-уведомления реализованы, но не проверены end-to-end в этой сессии — рекомендуется проверить вручную после подключения реальной БД и HTTPS-домена.

При тестировании этой сессии была найдена и исправлена гонка состояний: фронтенд (`ChatWindow`/`ChatList`) мог не успеть подписаться на сокет-события при прямой загрузке страницы чата, а backend регистрировал обработчики сокета уже после запроса к БД — оба места правились так, чтобы сообщения и события гарантированно доставлялись сразу, без необходимости обновлять страницу.

## Известные ограничения MVP

- Загруженные файлы хранятся на диске backend — на большинстве бесплатных хостингов (Render, Railway) диск эфемерный (файлы теряются при редеплое). Для надёжного хранения стоит перейти на S3-совместимое хранилище (например, Cloudflare R2) — см. раздел деплоя ниже
- Push-уведомления требуют HTTPS в проде (на localhost работают благодаря "secure context" исключению для localhost)
- PWA service worker собирается только в production-сборке (`npm run build`); в dev-режиме (`npm run dev`) он не активен

## Деплой (чтобы приложением могли пользоваться другие люди) — бесплатно

Для небольшого круга пользователей (друзья, знакомые) следующая связка бесплатна без ограничения по времени: **Neon** (Postgres) + **Render** (backend-сервер) + **Vercel** (frontend). Код уже параметризован через переменные окружения и одинаково подходит для этой связки, Railway или любого другого хостинга — ничего в коде менять не нужно.

Нужны 4 вещи: код в GitHub, база данных, backend и frontend.

### 0. Залить код в GitHub

Эта папка сейчас не git-репозиторий. Один раз:

```
git init
git add .
git commit -m "initial commit"
```

Создайте пустой репозиторий на GitHub и запушьте:

```
git remote add origin https://github.com/<ваш-аккаунт>/<репозиторий>.git
git branch -M main
git push -u origin main
```

### 1. PostgreSQL на Neon

[neon.tech](https://neon.tech) → зарегистрироваться → Create Project. В Dashboard проекта на вкладке Connection Details скопируйте **Connection string** (обычный, не pooled — он проще и для такого масштаба вполне достаточен). Это и есть `DATABASE_URL`, понадобится дальше. Бесплатный тариф Neon не имеет срока действия; при простое база "засыпает" и автоматически просыпается при первом запросе (доля секунды задержки).

### 2. Backend на Render

[render.com](https://render.com) → зарегистрироваться через GitHub → New → Web Service → выбрать ваш репозиторий:
- **Root Directory**: `backend`
- **Runtime**: Node
- **Build Command**: `npm install`
- **Start Command**: `npm start`
- **Instance Type**: Free
- **Environment Variables**, задать:
  - `DATABASE_URL` — из шага 1 (строка от Neon)
  - `JWT_SECRET`, `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` — значения возьмите из `backend/.env`
  - `FRONTEND_ORIGIN` — адрес фронтенда с шага 3 (сначала можно поставить заглушку и вернуться сюда после)
  - `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`, `CLOUDINARY_API_SECRET` — из [cloudinary.com](https://cloudinary.com) (бесплатный тариф), со страницы Dashboard. Без них загруженные файлы (аватарки, фото, вложения) пишутся на диск Render и стираются при каждом передеплое — с этими переменными они хранятся в Cloudinary и переживают деплой.

При каждом деплое Render сам генерирует Prisma Client и накатывает миграции (`prisma migrate deploy` уже встроен в `npm start`) — вручную ничего дополнительно выполнять не нужно. После деплоя Render выдаст домен вида `https://your-backend.onrender.com` — он нужен для шага 3.

Важно про бесплатный тариф Render: сервис "засыпает" после 15 минут без запросов и просыпается заново ~30–60 секунд при следующем обращении (первое сообщение после паузы придёт с задержкой, дальше работает как обычно, пока не наступит следующий простой). Для чата в кругу друзей это обычно не мешает.

### 3. Frontend на Vercel

[vercel.com](https://vercel.com) → Add New Project → выбрать тот же GitHub-репозиторий:
- **Root Directory**: `frontend`
- Framework Preset определится сам как Vite (`npm run build`, папка `dist`)
- **Environment Variables**: добавить `VITE_API_URL` = адрес backend из шага 2, без слэша на конце (например `https://your-backend.onrender.com`)

После первого деплоя Vercel выдаст домен вида `https://your-app.vercel.app`.

### 4. Досвязать backend и frontend

Вернитесь в Render (шаг 2) и обновите переменную `FRONTEND_ORIGIN` на адрес Vercel из шага 3 — иначе backend будет отклонять запросы с фронтенда (CORS). После сохранения Render передеплоит сервис автоматически.

### Готово

Откройте домен фронтенда (Vercel) — приложением может пользоваться кто угодно, у кого есть ссылка.

### Про хранение файлов (аватары, вложения)

Backend хранит загруженные файлы прямо на своём диске (`backend/uploads/`). И на Render, и на Railway этот диск **эфемерный** — при каждом передеплое (в том числе после "засыпания" на Render, если это full redeploy, а не просто пробуждение процесса) содержимое папки может обнуляться, и старые аватары/фото станут недоступны. Для маленького круга пользователей это не критично, но если захочется, чтобы файлы не терялись — дайте знать, вынесу загрузку файлов на бесплатное S3-совместимое хранилище (например, Cloudflare R2, 10 ГБ бесплатно), это отдельная небольшая доработка `backend/src/routes/upload.js`.
